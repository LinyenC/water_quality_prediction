# 当前模型与代码实现梳理

## 1. 项目当前到底是什么

当前这个项目不是“一个单独的大模型”，而是一套完整的 `HydroTail` 实验框架。

它服务的核心问题是：

> 在美国大陆多站点、日尺度、断续观测、缺测明显、目标长度不一致的情况下，如何预测 `specific_conductance` 和 `turbidity`，并同时分析尾部高值风险与未见站点泛化能力。

这套框架现在已经具备下面几类能力：

1. 支持真实目录式数据输入，而不只是手工整理好的平铺表格。
2. 支持通过 `dataset_root` 统一指定数据根目录，也支持分别指定 `attributes_root`、`time_series_root`、`wq_root`。
3. 支持断续站点记录和目标长度短于协变量长度的情况。
4. 支持多任务部分标签训练，也就是一行样本可以只带 `conductance` 或只带 `turbidity`。
5. 支持表格基线、深度表格模型、序列模型三类路线。
6. 支持图信息以三种方式进入 `torch_tail`：
   - `none`
   - `neighbor_stats`
   - `gnn`
7. 支持点预测、分位数预测、超阈值概率预测三类输出，其中点预测统一由 `q0.5` 导出。
8. 支持为 `torch_tail`、`seq_tcn_tail`、`seq_transformer_tail` 显式配置 `device`，可以运行在 `cpu` 或 `cuda`。
9. 支持 `unseen_station_and_future` 这类时空联合外推切分，并额外保存早期/后期分析切片。

当前主线可以概括为：

> 以 `torch_tail` 为主深度表格模型，以 `linear_tail` 和 `gbdt_tail` 为稳健基线，以 `seq_tcn_tail` 和 `seq_transformer_tail` 为序列扩展，同时把图信息注入方式收敛成 `torch_tail` 内部的一个可选设计。

## 3. 数据读取和组织

### 3.1 两种数据入口

当前框架支持两种入口：

1. 平铺表格入口
   - `data/dynamic_daily.csv`
   - `data/static_attributes.csv`
2. 目录式入口
   - 可以通过 `dataset_root` 统一解析
   - 也可以分别写 `attributes_root`
   - `time_series_root`
   - `wq_root`

不管走哪一种，最后都会变成：

- `dynamic_df`
- `static_df`

后续特征工程和训练逻辑是统一的。

### 3.2 目录式数据入口的当前约定

在当前真实配置 `configs/dataset_bundle_experiment.yaml` 中：

- 静态属性可以来自 `dataset/attritubes`，也可以来自单独配置的 `attributes_root`
- 动态驱动可以来自 `dataset/time series`，也可以来自单独配置的 `time_series_root`
  - `streamflow`
  - `daymet`
  - `PE`
  - `LAI`
  - `SWE_SNODAS`
- 水质目标可以来自 `dataset/WQ`，也可以来自单独配置的 `wq_root`

`dataset_bundle.py` 会把这些来源统一整理成：

- `station_id`
- `date`

这两个主键下的动态表。

### 3.3 当前 WQ 支持哪些格式

现在已经可以直接读取：

- `dataset/WQ/SEC/*.csv`
- `dataset/WQ/Tur/*.csv`

同时也兼容：

- `dataset/WQ/wq_observations.csv`
- 旧目录格式如 `Paired_EC` / `Paired_Tur`

如果未来又回到只有 MATLAB 文件的场景，需要注意：

- 旧的 `Paired_EC/*.mat`
- 旧的 `Paired_Tur/*.mat`

仍然是 MATLAB `table/timetable` 的 opaque MCOS 格式，当前 loader 不能直接解开，仍然需要先导出成 `csv`。

### 3.4 当前本地与服务器路径是怎么分开的

当前代码已经支持“同一个配置体系下，数据位置和运行位置分离”：

- Windows 本地实验可以把：
  - `attritubes` 放在一个目录
  - `time_series` 放在另一个目录
  - `WQ` 放在第三个目录

这意味着当前框架已经不要求“代码目录和数据目录必须在同一个根下面”。

## 4. dynamic 数据在代码里是怎么处理的

### 4.1 先做站点内日尺度对齐

`hydrotail/data.py` 先对每个站点做这几步：

1. 按 `station_id + date` 排序
2. 合并同一天的重复记录
3. 在该站点的起止日期之间按日补齐完整时间轴

这样做的作用是：

- 断档会被显式表示成 `NaN`
- 不会把相隔很多天的观测误认为连续时间步
- 后续 lag 和 rolling 都是在“完整日轴”上做，而不是在稀疏记录上做

### 4.2 保留缺测与空档信息

对每个原始时间序列变量，框架会额外保留缺测相关特征，例如：

- `xxx_observed`
- `is_original_observation`
- `is_gap_filled`
- `any_dynamic_observed`

这意味着模型能区分三种情况：

1. 真实观测到的数值
2. 原本就缺测的数值
3. 因为补齐时间轴而产生的“空档位置”

### 4.3 再生成监督学习特征

在补齐后的日轴上，`build_model_frame()` 会继续生成：

- lag 特征
  - `lag_1`
  - `lag_3`
  - `lag_7`
  - `lag_14`
  - `lag_30`
- 滚动统计
  - `roll_mean`
  - `roll_std`
- 季节特征
  - `doy_sin`
  - `doy_cos`
  - `month`

这些特征不仅对动态驱动做，也对目标变量自身的历史值做。

### 4.4 未来目标怎么构造

对每个目标，代码都会构造：

- `target_conductance_h{horizon}`
- `target_turbidity_h{horizon}`

本质就是：

- 用当前时刻和过去的信息
- 预测未来 `horizon` 天的目标值

这里有一个非常关键的实现细节：

- 现在不是要求两个目标都必须有未来标签
- 而是只要求“至少一个目标在未来存在观测”

也就是说：

- 一条样本可以只有 `conductance`
- 也可以只有 `turbidity`
- 也可以两个都有

这就是当前框架支持多任务部分标签训练的基础。

## 5. sequence 路线怎么处理

如果走序列模型，也就是：

- `seq_tcn_tail`
- `seq_transformer_tail`

则 `build_sequence_samples()` 会把站点-日期表转成：

- `lookback window -> sequence tensor`

### 5.1 sequence 输入包含什么

序列输入一般包含：

- 动态驱动变量
- 目标变量自身历史
- `is_gap_filled`
- 季节项

同时，每个窗口都会额外生成：

- `sequence_values`
- `sequence_masks`

这里的 `mask` 很重要，因为它告诉模型：

- 哪些位置是真实观测
- 哪些位置只是对齐后留下的缺测空位

### 5.2 为什么断续站点也能做序列建模

原因不在于“先插值补满”，而在于：

1. 先对齐到完整日轴
2. 保留缺测为 `NaN`
3. 再把数值和 mask 一起送进序列模型

所以当前序列模型实际上看到了“值 + 缺测模式”，而不是只看一个被静默填补过的序列。

### 5.3 序列样本什么时候会被跳过

如果一个窗口里几乎全是缺测，`build_sequence_samples()` 会直接跳过这个窗口。

具体由配置控制：

- `sequence.lookback_window`
- `sequence.min_history_ratio`

这样可以防止过于稀疏的窗口把序列模型训练带崩。

### 5.4 `sequence_tail` 和图信息现在是什么关系

当前序列路线现在也已经像 `torch_tail` 一样暴露了：

- `graph_backend = none`
- `graph_backend = neighbor_stats`
- `graph_backend = gnn`

也就是说，`sequence_tail` 现在不只是“间接吃到图特征”，而是已经可以直接切三种图信息使用方式。

更准确地说：

- `none`
  - 只做时间编码 + 普通静态特征融合
- `neighbor_stats`
  - 保留 `graph_neighbor_*` 这类预计算邻居统计特征，并通过静态分支融合
- `gnn`
  - 去掉这些预计算邻居统计特征
  - 先做 `TCN/Transformer` 时间编码
  - 再按样本最后一天的日期做 same-day 图传播

所以现在 `sequence_tail` 已经可以直接做图后端比较，只是它的 `gnn` 仍然属于“时序先编码、同日图再传播”的轻量实现，而不是完整的时空图网络。

## 6. 图模块现在怎么组织

### 6.1 `graph.py` 先构造静态相似图

`hydrotail/graph.py` 会基于静态流域属性做最近邻搜索，输出两类东西：

1. 邻居统计特征
   - `graph_neighbor_mean_xxx`
   - `graph_neighbor_mean_distance`
   - `graph_neighbor_count`
2. 边表
   - `source_station`
   - `neighbor_station`
   - `distance`

这一步是基于静态属性的，所以图结构本身不是按每天重建的。

### 6.2 GNN 不再是独立模型

之前思路里曾经把 GNN 作为一个单独模型路线，现在已经收口成：

- `torch_tail` 内部的一个可选图后端
- `sequence_tail` 内部的一个可选图后端

配置方式统一成：

- `graph_backend = none`
- `graph_backend = neighbor_stats`
- `graph_backend = gnn`

也就是说，当前“图信息怎么用”已经变成跨模型族共享的一条 ablation 维度，而不是多个完全不同模型族并列。
### 6.3 三种 `graph_backend` 的区别

#### `none`

- 不使用邻居统计特征
- 不使用图传播
- 纯表格深度模型

#### `neighbor_stats`

- 使用 `graph.py` 预先算好的邻居统计特征
- 这些特征和其他表格特征一起进入共享 MLP
- 本质上还是“图启发的特征工程”

#### `gnn`

- 会先把 `graph_neighbor_*` 这类预计算邻居统计特征从输入中移除
- 再直接使用边表 `edge_df`
- 在模型内部做 same-day 图传播
- 对 `torch_tail` 来说，是“表格特征 -> 图传播”
- 对 `sequence_tail` 来说，是“时间编码 -> 图传播”
### 6.4 当前 GNN 后端具体怎么工作

`hydrotail/models/graph_backends.py` 当前做的是一种轻量 same-day 图建模：

1. 按日期把样本分组
2. 每个日期取当天出现的站点作为一个子图
3. 用静态相似图里已有的边关系，在当天子图上构造邻接矩阵
4. 节点特征来自当天样本在各自主干里的编码表示
   - 对 `torch_tail` 来说，是当天该行的表格特征编码
   - 对 `sequence_tail` 来说，是 lookback window 经过 `TCN/Transformer` 后得到的时间表示
5. 经过轻量图卷积后，再接 quantile head 和 event head

所以当前的 `gnn` 不是完整时空 GNN，也不是 Graph Transformer。

更准确地说，它是：

- 时间信息来自 lag / rolling / seasonal 特征，或来自 `TCN/Transformer` 编码后的窗口表示
- 空间信息来自 same-day 图传播

这非常适合当前阶段，因为它正好能回答一个很实际的问题：

> 相比“预先把邻居关系压成统计特征”，直接做图传播是否有增益。

## 7. 模型家族当前怎么分

### 7.1 表格基线

在 [sklearn_tail.py](F:\research\water quality prediction\hydrotail\models\sklearn_tail.py) 里有两个基线：

- `LinearTailModel`
- `GBDTTailModel`

它们的共同特点是：

- 以表格特征为输入
- 每个目标分别训练
- 输出多个 quantiles
- 输出 exceedance probability
- 最终点预测由最接近 `0.5` 的 quantile 导出

如果某个目标在训练集中一个样本都没有，它会：

- 跳过该目标训练
- 预测时对该目标输出 `NaN`

所以它们已经兼容部分标签场景。

### 7.2 主深度表格模型 `torch_tail`

在 [torch_tail.py](F:\research\water quality prediction\hydrotail\models\torch_tail.py) 里，`TorchTailModel` 是当前最重要的主模型。

它分成两种主干形式：

1. 普通共享 MLP
2. 图主干 `GraphBackbone`

但不管哪种主干，输出结构都一样：

- 每个目标一个 quantile head
- 每个目标一个 event head

另外，`torch_tail` 现在已经支持通过配置指定设备：

- `device = auto`
- `device = cpu`
- `device = cuda`
- `device = cuda:0`
- `device = cuda:1`

所以它已经可以在 Linux 服务器上显式使用 GPU，而不是只能走 CPU。

### 7.3 序列模型

在 [sequence_tail.py](F:\research\water quality prediction\hydrotail\models\sequence_tail.py) 里有：

- `seq_tcn_tail`
- `seq_transformer_tail`

它们共享同一套尾部输出逻辑，只是时间编码器不同：

- `TCN` 用因果卷积块编码
- `Transformer` 用自注意力编码

现在序列模型也已经支持和表格路线相同的三种图后端：

- `none`
- `neighbor_stats`
- `gnn`

因此序列路线不再只是“时间建模 + 静态特征融合”，而是已经可以直接比较：

- 纯时间建模
- 时间建模 + 邻居统计特征
- 时间建模 + same-day 图传播

同时，序列路线现在也和 `torch_tail` 一样支持 `device` 配置，因此：

- `seq_tcn_tail`
- `seq_transformer_tail`

都已经能显式切到 `cuda:0 / cuda:1`。

## 8. 输出层与损失函数

### 8.1 现在不再学习独立 point head

当前实现已经移除了单独学习的 `point head`。

现在模型学习的是：

- `quantile head`
- `event head`

保存结果时的 `point prediction` 统一由 `q0.5` 导出。

这样做的好处是：

- 避免 `point head`、`quantile head`、`event head` 之间出现明显不一致
- 让点预测和分位数输出更统一

### 8.2 每个目标现在输出什么

对每个目标，当前预测输出都包含：

1. `point`
   - 实际来自 `q0.5`
2. `quantiles`
   - 例如 `0.1`
   - `0.5`
   - `0.9`
   - `0.95`
   - `0.99`
3. `exceedance_probability`
   - 即超过阈值的概率

### 8.3 当前损失由什么组成

对深度模型而言，核心损失包括：

- `quantile loss`
- `binary cross-entropy`
- 非负边界惩罚

其中：

- `quantile_weights` 可以让高分位得到更高权重
- `tail_weight_multiplier` 会给尾部样本更高训练权重

### 8.4 部分标签是怎么进入 loss 的

这是当前实现中最关键的一点之一。

在 `torch_tail` 和序列模型里，每个目标都会单独构造：

- `valid_mask = torch.isfinite(target)`

然后：

- quantile loss 只在该目标有标签的位置上算
- event loss 只在该目标有标签的位置上算
- boundary loss 也只在该目标有标签的位置上算

所以当前多任务其实不是要求两个目标同步出现，而是：

> 共享 backbone，但每个目标在自己的可观测位置上单独反向传播。

## 9. 切分现在到底是怎么做的

### 9.1 支持三种切分策略

在 [splits.py](F:\research\water quality prediction\hydrotail\splits.py) 中，支持：

- `unseen_station`
- `time`
- `unseen_station_and_future`

### 9.2 默认主策略是 `unseen_station_and_future`

当前默认配置里主策略是：

- `unseen_station_and_future`

但要注意，它的实现不是一个普通的二维网格切分，而是：

- `train`
  - 只取 `train_station + early`
- `valid`
  - 只取 `valid_station + middle`
- `test`
  - 只取 `test_station + late`

所以当前 `train / valid / test` 实际上分别对应：

1. 熟站点早期
2. 验证站点中期
3. 未见站点后期

这是一种比较强的时空联合外推设置。

### 9.3 额外分析切片是什么

除了 `split` 以外，代码还会额外标记：

- `train_station_early`
- `train_station_late`
- `test_station_early`
- `test_station_late`

这些切片不一定都会作为正式 `train / valid / test` 使用，但会在评估阶段尽量单独保存。

这里需要注意一个现实细节：

- 某些切片可能为空
- 尤其当你的 `EC` 和 `Tur` 站点重叠很少时，某些 group 里可能只剩单一目标

所以当前框架支持这些分析切片，但并不保证每个数据子集都一定非空。

## 10. `train.py` 现在到底怎么跑完整实验

`hydrotail/train.py` 当前完整流程可以分成下面几步：

1. 读取配置
2. 读取数据
3. 构造 `frame`
4. 构造图边 `edge_df`
5. 按策略切分
6. 基于训练集计算每个目标的 exceedance 阈值
7. 根据阈值构造训练样本权重
8. 按模型类型分支训练
9. 先保存本次实验配置快照
   - `experiment_config.yaml`
10. 对所有可用评估切片保存：
   - `metrics.json`
   - `predictions.csv`
   - `model.bin`

### 10.1 阈值怎么来的

阈值不是手工固定写死的，而是：

- 对训练集目标值取配置里的高分位

比如当前默认配置里：

- `conductance` 用 `0.95`
- `turbidity` 用 `0.9`

如果某个目标在训练集根本没有观测值，那么该目标阈值会变成 `NaN`。

### 10.2 样本权重怎么来的

`_compute_sample_weights()` 会判断每行样本是否触发任一目标的尾部事件：

- 如果触发，就提高该行权重
- 如果没有触发，就保留基础权重

所以这一步是“按行重权”，不是直接按单个目标单独采样。

### 10.3 不同模型怎么训练

#### `linear_tail` 和 `gbdt_tail`

这两个模型会把：

- `train`
- `valid`

拼起来一起训练，然后对所有评估 group 预测和保存结果。

也就是说，它们不做深度模型式的 early stopping。

#### `torch_tail`

`torch_tail` 会显式使用：

- `train_frame`
- `valid_frame`

做训练与验证分开优化。

如果 `graph_backend = gnn`，还会额外拿到：

- `edge_df`
- `station_col`
- `date_col`

用于构造同日图快照。

#### 序列模型

序列模型会先把 `frame` 变成 `SequenceSamples`，再按：

- `train`
- `valid`
- `test`
- 其余分析 group

分别切 bundle 来训练和评估。

### 10.4 当前保存了哪些结果

每个 horizon 下会保存：

- `experiment_config.yaml`
- `split_metadata.json`
- `thresholds.json`
- `graph_edges.csv`（如果图启用）
- 各模型目录

另外，在整次 run 的根输出目录下还会额外保存一份：

- `experiment_config.yaml`

每个模型目录下又会保存：

- `train_metrics.json`
- `valid_metrics.json`
- `test_metrics.json`
- 额外 group 的 `*_metrics.json`
- 对应的 `*_predictions.csv`
- `model.bin`

所以当前框架不是只输出一张汇总表，而是会把每个模型、每个切片的详细结果都落盘。

## 13. 当前版本的边界和限制

### 13.1 旧 MATLAB timetable 文件仍然不能直接用

如果未来没有 `csv`，只有旧的 `Paired_EC/*.mat` 和 `Paired_Tur/*.mat`，当前 loader 还是没法直接吃。

### 13.2 当前 `gnn` 仍然是轻量版本

现在的 GNN：

- 不是完整的时空 GNN
- 不是 Graph Transformer
- 虽然已经能接在 `TCN` 或 `Transformer` 序列编码器后面，但仍然只是“时间先编码、同日图再传播”的轻量耦合

它的定位更像：

- 轻量图后端
- 用来和邻居统计路线做第一轮比较
### 13.3 当前切分策略比较“强”

默认的 `unseen_station_and_future` 实现是：

- `train_station + early`
- `valid_station + middle`
- `test_station + late`

这是一种很强的时空联合外推设置。

它的优点是难度高、研究味重。

它的代价是：

- 有些数据子集可能非常小
- 某些目标在某些 split 里可能根本没有观测值

### 13.4 当前部分标签虽然已经支持，但真实数据分布仍会影响结果解释

代码已经支持部分标签训练，但如果你的真实数据里：

- `EC` 和 `Tur` 在站点层面重叠很少

那么某些 split 或某些分析切片中可能会出现：

- `conductance` 可以评估
- `turbidity` 没有测试观测

这不是代码错误，而是数据覆盖结构本身带来的结果。

### 13.5 当前还有 pandas 特征构造性能问题

`data.py` 现在在逐列插入 lag 和 rolling 特征时，可能会出现 pandas 的 fragmentation warning。

它目前不影响结果正确性，但会影响速度。

### 13.6 当前真实数据首轮运行瓶颈在数据装载和拼接

截至 `2026-03-22`，服务器上的第一轮真实运行已经启动，但在进入深度模型训练前，先花了很长时间做数据读取和整理。

当前已确认的数据规模大致是：

- `time_series` 约 `5034` 个文件
- `WQ` 约 `1435` 个文件

而当前 `dataset_bundle.py` 的实现是逐文件读取后再 `concat`，因此首轮运行的第一个主要瓶颈更像是：

- 文件级 I/O
- pandas 拼接
- 特征构造前的数据预整理

而不是 GPU 配置本身。

### 13.7 当前训练日志还不够“可观测”

现在训练入口在真正进入某个 horizon / model 之前，没有显式打印进度日志。

所以当运行还停留在：

- `load_datasets()`
- `build_model_frame()`
- 早期基线训练

这些阶段时，`train.log` 可能仍然是空的。

这不会影响结果正确性，但会让长任务看起来像“卡住了”，后面更适合补：

- 数据加载开始/结束日志
- horizon 开始日志
- model 开始日志

## 14. 现在最自然的实验推进顺序

如果继续推进实验，我建议按下面顺序走：

1. 先用真实数据跑第一轮表格基线
   - `linear_tail`
   - `gbdt_tail`
2. 再跑主深度表格模型
   - `torch_tail` with `graph_backend=none`
   - `torch_tail` with `graph_backend=neighbor_stats`
   - `torch_tail` with `graph_backend=gnn`
3. 再补序列路线
   - `seq_tcn_tail` with `graph_backend=none / neighbor_stats / gnn`
   - `seq_transformer_tail` with `graph_backend=none / neighbor_stats / gnn`
4. 分析这些维度
   - `test`
   - `train_station_early`
   - `train_station_late`
   - `test_station_early`
   - `test_station_late`
   - 尾部高值指标
   - `neighbor_stats` vs `gnn`
   - 表格路线 vs 序列路线

如果当前数据的 `EC/Tur` 重叠仍然不够，那更稳的顺序是：

1. 先分别验证两个目标各自能不能稳定训练和评估
2. 再做联合多任务解释

## 15. 当前代码可以做哪些对比实验

### 15.1 模型家族对比

当前代码最直接支持的第一类对比，是不同模型家族之间的比较：

- `linear_tail`
- `gbdt_tail`
- `torch_tail`
- `seq_tcn_tail`
- `seq_transformer_tail`

这组对比能回答的是：

> 传统线性方法、树模型、深度表格模型和序列模型，在当前日尺度多站点任务上谁更稳、谁更强。

### 15.2 `torch_tail` 内部的图信息注入方式对比

这是当前框架最完整、也最推荐优先做的一组方法对照：

- `torch_tail + none`
- `torch_tail + neighbor_stats`
- `torch_tail + gnn`

这组对比最核心的问题是：

> 完全不用图信息、只用预计算邻居统计特征、以及直接做 same-day 图传播，这三种方式到底差多少。

如果从论文角度看，这也是当前最清晰的一组 ablation。

### 15.3 时间建模方式对比

在时间编码层面，当前代码已经支持：

- 表格时间建模
  - 通过 lag / rolling / seasonal 特征间接编码时间
- `TCN` 序列建模
- `Transformer` 序列建模

所以可以做两层比较：

1. 表格路线 vs 序列路线
2. `seq_tcn_tail` vs `seq_transformer_tail`

这组实验适合回答：

> 在当前这种断续日序列和部分标签场景下，显式 lookback window 建模是否真的优于 lag/rolling 表格特征。

### 15.4 序列路线当前能做的图相关对比

现在 `sequence_tail` 也已经支持直接的图后端切换，所以可以做完整的序列路线图相关比较：

- `seq_tcn_tail + none`
- `seq_tcn_tail + neighbor_stats`
- `seq_tcn_tail + gnn`
- `seq_transformer_tail + none`
- `seq_transformer_tail + neighbor_stats`
- `seq_transformer_tail + gnn`

因此当前代码不再只是让序列模型做“时间编码器对比”，而是已经可以进一步回答：

- 同一种时间编码器是否真的从图信息里获益
- `neighbor_stats` 和 `gnn` 在序列路线里谁更有效
- 图信息增益是否依赖编码器类型
### 15.5 切分场景对比

当前默认切分是 `unseen_station_and_future`，并且额外保存了四类分析切片：

- `train_station_early`
- `train_station_late`
- `test_station_early`
- `test_station_late`

所以你现在不仅能看常规：

- `train`
- `valid`
- `test`

还可以进一步比较：

- 熟站点早期 vs 熟站点后期
- 未见站点早期 vs 未见站点后期
- 熟站点 vs 未见站点
- 早期 vs 后期

这组分析很适合回答：

> 模型退化到底主要来自时间外推，还是空间泛化，或者两者叠加。

### 15.6 整体样本与尾部样本对比

当前输出和评估保存方式，也支持做“整体表现”和“尾部表现”的对比。

可以重点看：

- 全样本 `MAE / RMSE / R2`
- 各 quantile 的 `pinball loss`
- exceedance 相关指标
- 高值样本上的误差表现

因此当前框架不只适合回答“平均上谁更准”，也适合回答：

> 谁对极端高浊度和异常高比电导更敏感、更稳。

### 15.7 单目标与双目标覆盖差异下的现实实验

由于当前代码支持 target-specific NaN masking，所以在真实数据覆盖不完全时，可以自然出现下面几类实验状态：

- 两个目标联合训练、联合评估
- 两个目标联合训练，但某些切片只评估其中一个目标
- 某一轮实验事实上退化成单目标评估

这不是代码错误，而是当前数据覆盖结构下的一种正常实验形态。

因此在写实验设计时，最好明确区分：

- 代码能力上支持双目标多任务
- 数据现实上不一定每个 split 都能同时完整评估两个目标

### 15.8 当前最推荐优先跑的对比组合

如果按当前代码成熟度排序，我最推荐优先跑下面四组：

1. 基线组
   - `linear_tail`
   - `gbdt_tail`
2. 主方法表格组
   - `torch_tail + none`
   - `torch_tail + neighbor_stats`
   - `torch_tail + gnn`
3. 序列 TCN 组
   - `seq_tcn_tail + none`
   - `seq_tcn_tail + neighbor_stats`
   - `seq_tcn_tail + gnn`
4. 序列 Transformer 组
   - `seq_transformer_tail + none`
   - `seq_transformer_tail + neighbor_stats`
   - `seq_transformer_tail + gnn`

这四组已经足够构成一套比较完整的方法对照框架。

## 2.7 2026-03-23：数据入口加速与 smoke 改造

这轮新增了三块关键能力：

- `coverage-aware smoke subset`
  - 当配置了 `station_limit` 时，不再简单取前 N 个站点
  - 现在会优先选择同时覆盖多个目标的站点子集，用来改善 `conductance + turbidity` 的 smoke 验证稳定性

  - bundle 读取后和特征构造后都会对数值列做 `float32` 下采样

## 3.6 现在 `dataset_bundle` 比之前多了什么

`hydrotail/dataset_bundle.py` 现在除了原来的目录读取外，还多了：

- 针对内存的基础优化

## 4.5 为什么现在优先缓存 bundle，而不是缓存最终特征表

当前实现刻意把缓存放在“bundle 合并后、特征工程前”这一层。

原因是：

- 如果缓存最终特征表，那么一旦修改：
  - `lags`
  - `rolling_windows`
  - `horizon`
  - `min_non_null_feature_ratio`
  就必须整表重建
- 如果缓存 bundle 层：
  - 改模型
  - 改 batch size
  - 改 split
  - 改大部分训练参数
  通常都不需要重做原始 CSV 解析和目录合并

所以当前缓存策略更像：

- 第一层：加速原始目录数据读取
- 第二层：保留后续实验配置的灵活性

## 8.3 Linux smoke 现在为什么改成 `unseen_station`

主实验配置仍然保留：

- `unseen_station_and_future`

但 Linux smoke 配置现在改成：

- `unseen_station`

原因不是研究问题变了，而是 smoke 的目标不同：

- smoke 要先验证“目录读取 + 缓存 + 特征 + 训练 + 输出”整条链路是否跑通
- 对于像 `turbidity` 这样更稀疏的目标，小站点子集在 `unseen_station_and_future` 下可能出现“目标整体存在，但 train 分区刚好没有标签”的情况
- 因此 smoke 先用 `unseen_station` 做流程验证，主实验再回到严格的联合时空外推切分

## 10.2 2026-03-23 服务器验证结果

本轮在 Linux 服务器上的真实验证已经确认：

- `coverage_aware` 的 32 站点 smoke 子集能够同时覆盖 `conductance` 和 `turbidity`
- 首次冷启动会生成：
  - `attributes.parquet`
  - `time_series.parquet`
  - `wq_observations.parquet`

仍需注意：

- 目前更准确的说法是：
  - smoke 验证链路已显著稳定
  - 主实验切分下的稀疏目标可学习性，还需要后续专门评估

补充说明（2026-03-23，fast smoke mode）：

- 当前 Linux smoke 配置已进一步收敛为只跑 `torch_tail`
- 这样做不是放弃 baseline，而是把 baseline 与 smoke 分离：
  - smoke 负责快速验证“缓存命中 + 多目标标签 + GPU 训练链路”

补充说明（2026-03-23，formal static set v1）：

当前正式实验配置中的静态属性已经从“15 个核心属性”扩展到“15 个核心属性 + 20 个新增属性”。

本轮一次性加入的新增属性分为五组：

- 水文分布：`Q50`、`Qstd`、`Q10`
- 气候分布：`P50`、`P95`、`PEstd`
- 植被与生物量：`LAIstd`、`agBiomass`
- 灌溉 / 供水 / 水库调蓄：`iaFmean`、`IWUmean`、`IWU_SI`、`RS_mean`、`RS_std`、`PSW_SW_mean`、`PSW_SW_si`、`PSW_GW_mean`、`PSW_GW_si`
- 土壤持水与质地：`Wsat`、`WPnt`、`vf_clay_s`

这版选择的原则不是“相关性最高的全收”，而是：

- 保留当前已用核心属性，保证和旧结果可比
- 只加入同时满足“相关性较强 + 物理含义清楚 + 不明显重复”的变量
- 暂时不把这些新增属性直接加入图构造特征；图特征仍保持现有较稳的一组，后面可以单独做 ablation

## 10.3 2026-03-23 温度动态变量清理

当前正式与 smoke 配置已改为保留 `tmin`、`tmax`，移除 `air_temp`。

这样做的原因是：

- `air_temp` 在当前 bundle loader 中本质上由 `tmin` 和 `tmax` 的均值生成
- 若三者同时进入特征工程，会一起扩展出 lag、rolling、observed flag 等派生列，造成明显冗余
- 因此当前主线实验默认只保留 `tmin` 与 `tmax`

为了保留后续 ablation 的灵活性，代码层面仍支持在配置显式请求 `air_temp` 时再按需生成。

## 10.4 2026-03-23 模型帧性能优化

在全量 Linux 正式实验里，原始目录读取与 source cache 构建已经成功完成，新的主要瓶颈转移到了 `build_model_frame`。

这次针对该阶段做了两类改动：

- 把 lag / rolling 特征从逐列 `frame[col] = ...` 插入，改成按批次组装后再拼接
- 为模型帧构造补充了更细的日志，包括：
  - daily frame 行数
  - target-eligible rows
  - 每个原始时序变量的 temporal feature 构造进度
  - static merge 后的行列规模

这样做的目标是减少 pandas DataFrame 碎片化，并让下一轮服务器重跑可以更准确地判断剩余瓶颈是否还在特征工程本身。

## 10.5 2026-03-23 WQ cache 清理与实验配置拆分

数据审计之后，当前实现又补了两项直接面向下一轮实验的改动。

### 1. WQ cache 现在会在缓存阶段丢弃“双目标全空”行

当前 `dataset_bundle` 在构建或读取 WQ bundle 时，会对配置中声明的目标列执行一条统一规则：

- 如果某一行在所有目标列上都为 `NaN`
- 则该行不会保留到新的 `wq_observations.parquet` component cache 中

这样做的目的不是改变有效观测，而是清掉大量对训练与评估都没有贡献的结构性空行。为了避免新旧缓存混用，bundle cache 的 `schema_version` 已进一步提升。

### 2. 主线配置已拆成 conductance-only 正式实验 + turbidity later-era pilot

新增了两份 Linux 配置：

- `configs/dataset_bundle_linux_formal_conductance.yaml`
  - 只保留 `conductance`
  - 只跑 `torch_tail`
  - 作为下一轮主正式实验配置
- `configs/dataset_bundle_linux_turbidity_later_era_pilot.yaml`
  - 只保留 `turbidity`
  - `data.date_filter.start_date = 2000-01-01`
  - 作为后续稀疏目标 pilot 配置

同时，当前主线 bundle 配置中已统一去掉：

- 动态 `lai`、`swe`
- 静态 `LAImean`、`LAIstd`、`SWEmean`
- 图构造中的 `SWEmean`

这样做的依据不是“这些变量永远没用”，而是：

- 在当前 next-day 设定和原 formal split 下，它们对 train 没有有效支持
- 与其保留为结构性全空特征，不如先把主线实验收敛到真正有训练支持的变量集合

## 10.6 2026-03-24 去除目标历史特征与 tail_count 修正

在检查 `conductance-only` 正式实验结果后，当前实现又补了两项重要修正。

### 1. 模型输入默认不再使用目标历史值

当前 `build_model_frame` 与 `build_sequence_samples` 新增了一个统一开关：

- `features.include_target_history_features`

主线 bundle 配置现已统一设为：

- `false`

这意味着：

- `specific_conductance` / `turbidity` 仍然会保留在数据集中，用于构建未来目标标签
- 但它们不会再自动进入模型输入
- 也不会再衍生出对应的 `*_observed`、lag、rolling 等特征

这样做的目的，是让当前主线 conductance 实验更接近“在完全没有 conductance 监测的流域上，仅依赖气象、流量与静态流域属性进行预测”的设定。

### 2. `tail_count` 指标命名问题已修正

之前结果文件里会出现：

- `tail_count = 0`
- 同时又有真实的 `tail_tail_count`

根因是评估阶段对 tail 子集指标统一追加了 `tail_` 前缀，导致 `tail_count` 被再次包了一层。

当前修正后：

- `tail_count` 将直接表示真实的 tail 样本数
- `tail_mae`、`tail_rmse`、`tail_r2` 保持不变
- 不再生成多余的 `tail_tail_count`

## 10.8 2026-03-24 新增 `seq_retrieval_prototype_tail`

当前代码已经新增一条新的 strict-ungauged sequence 路线：

- `seq_retrieval_prototype_tail`

这条路线的定位不是普通的时间编码器替换，而是显式引入：

- donor-basin retrieval
- learnable prototype memory
- local / donor / prototype 的 gated fusion

目前实现的是可跑的 `v1`：

- 仍然复用当前的 `SequenceEncoder`
- 当前 donor memory 是 station-level，而不是 sample-level
- 当前优化版会缓存 donor candidate 索引，并按 `memory_refresh_interval` 重建 station memory bank，而不是每个 epoch 都重建
- target sample 用自己的 exogenous sequence 和 static attributes 形成 query
- 再从 source-basin station memory 中取 top-k donor，并和 prototype token 一起融合
- 输出头仍然沿用 quantile + exceedance
- 训练阶段现在会记录每个 epoch 的 `train_loss / valid_loss / best_valid / wait`

当前推荐配置文件是：

- `configs/dataset_bundle_linux_formal_conductance_retrieval.yaml`
- `configs/dataset_bundle_linux_formal_conductance_retrieval_ablation.yaml`

其中 ablation 配置会把下面 4 个模型放到同一轮 run 里，以共享同一份 sequence samples：

- `seq_tcn_tail`
- `seq_retrieval_prototype_tail`
- `seq_retrieval_prototype_tail_nograph`
- `seq_retrieval_prototype_tail_noproto`

这条路线与论文主线是一致的，因为：

- 预测时不需要目标流域自己的 `specific_conductance` 历史
- donor knowledge 全部来自训练流域的 source memory bank
