# 当前模型与代码实现梳理

## 1. 当前版本的核心变化

现在这套 `HydroTail` 代码，已经从“多模型并列堆叠”收成了一个更清晰的实验框架。

和上一版相比，当前最关键的变化有 6 个：

1. **不再单独学习 point head**
   - 模型只学习 `quantiles` 和 `exceedance probability`
   - 对外导出的点预测统一使用 `q0.5`
2. **支持真实目录式数据入口**
   - 可以直接读取 `dataset/attritubes`
   - `dataset/time series`
   - `dataset/WQ`
3. **WQ 现在已经支持分站点 csv**
   - 当前可直接读取 `dataset/WQ/SEC/*.csv`
   - 当前可直接读取 `dataset/WQ/Tur/*.csv`
   - 同时仍兼容统一 `wq_observations.csv`
4. **显式处理断续时间序列和目标长度不一致**
   - 先按站点补齐日时间轴
   - 再做 lag / rolling / sequence window
   - 多任务训练允许每一行只带部分目标标签
5. **`unseen_station_and_future` 下增加四类分析切片**
   - `train_station_early`
   - `train_station_late`
   - `test_station_early`
   - `test_station_late`
6. **GNN 不再是独立模型**
   - 现在它是 `torch_tail` 的一个可选图后端
   - 也就是：`graph_backend = none | neighbor_stats | gnn`

这意味着当前框架的主线已经比较统一：

> 以 `torch_tail` 为主深度表格模型，以 `linear_tail` / `gbdt_tail` 为稳健基线，以 `seq_tcn_tail` / `seq_transformer_tail` 为序列扩展，并把图信息处理方式收敛成 `torch_tail` 内部的一个可选设计。

---

## 2. 现在代码服务的研究问题是什么

当前代码对应的问题可以概括成一句话：

> 在美国大陆多站点、日尺度、断续观测、缺测显著、目标序列长度不一致的环境下，如何同时预测 `specific_conductance` 和 `turbidity`，并重点分析新站点泛化与高值尾部风险。

这里面有三层任务：

1. **普通预测**
   - 未来 `horizon` 天的比电导和浊度预测
2. **尾部风险建模**
   - 不只预测中心值
   - 还要预测高分位和超阈值概率
3. **泛化分析**
   - 不只看 `train / valid / test`
   - 还要分别看熟站点/新站点、早期/后期

## 3. 代码目录和职责

### 3.1 入口与配置

- `hydrotail/train.py`
  - 训练总入口
  - 负责数据读取、特征构造、切分、阈值计算、训练、评估和结果保存
- `hydrotail/config.py`
  - 读取 YAML，并把相对路径解析为绝对路径
- `configs/default_experiment.yaml`
  - 平铺 `csv` 输入版本
- `configs/dataset_bundle_experiment.yaml`
  - 真实 `dataset/` 目录输入版本

### 3.2 数据层

- `hydrotail/dataset_bundle.py`
  - 读取 `dataset/attritubes`
  - 读取 `dataset/time series`
  - 读取 `dataset/WQ`
  - 合并成统一的动态表和静态表
- `hydrotail/data.py`
  - 每个站点按日补齐时间轴
  - 保留缺测位置
  - 构造 lag / rolling / seasonal 特征
  - 构造 sequence 的 lookback window

### 3.3 图与切分

- `hydrotail/graph.py`
  - 根据静态属性建立相似站点关系
  - 输出邻居统计特征和边列表
- `hydrotail/splits.py`
  - 负责 `unseen_station`
  - `time`
  - `unseen_station_and_future`
  - 以及四类额外分析切片

### 3.4 模型层

- `hydrotail/models/sklearn_tail.py`
  - `linear_tail`
  - `gbdt_tail`
- `hydrotail/models/torch_tail.py`
  - `torch_tail`
  - 以及内部 `graph_backend`
- `hydrotail/models/graph_backends.py`
  - GNN 后端使用的图卷积层
  - 同日子图快照构造
- `hydrotail/models/sequence_tail.py`
  - `seq_tcn_tail`
  - `seq_transformer_tail`

### 3.5 验证与说明

- `hydrotail/smoke_test.py`
  - 跑两套 smoke test
  - `neighbor_stats` 和 `gnn`
- `EXPERIMENT_SETUP.md`
  - 运行说明

## 4. 数据读取现在是怎么做的

### 4.1 两种数据入口

现在有两种入口：

1. **平铺表格入口**
   - `data/dynamic_daily.csv`
   - `data/static_attributes.csv`
2. **目录式入口**
   - `dataset/attritubes`
   - `dataset/time series`
   - `dataset/WQ`

两种入口最后都会转成：

- 一个动态表 `dynamic_df`
- 一个静态表 `static_df`

后面的特征工程和训练流程共用。

### 4.2 真实目录式数据是怎么组织的

目录式读取里：

- `attributes` 负责静态流域属性
- `time series` 负责动态驱动变量
  - 例如 `streamflow`
  - `daymet`
  - `PE`
  - `LAI`
  - `SWE_SNODAS`
- `WQ` 负责目标变量
  - 当前可直接读取 `SEC/*.csv`
  - 当前可直接读取 `Tur/*.csv`
  - 仍兼容旧的 `Paired_EC` / `Paired_Tur`

动态变量和水质目标都会按：

- `station_id`
- `date`

合并到统一的动态表里。

### 4.3 当前 WQ 的真实状态

现在 `dataset/WQ` 已经存在可直接使用的 csv 版本，所以：

- `dataset_bundle` 已经能直接解析
- 不再要求你先做统一 `wq_observations.csv`
- 也不再被旧 `.mat` 阻塞

不过要注意：

- 旧的 `Paired_EC/*.mat`
- `Paired_Tur/*.mat`

仍然还是 MATLAB `timetable` 的 opaque MCOS 格式。

所以当前更准确的说法是：

- **当前项目已经可以直接用 WQ csv 跑**
- **但如果未来回到 MAT-only 输入，仍然需要导出成 csv**

## 5. dynamic 数据现在怎么处理

这是当前代码里很关键的一层。

### 5.1 先补齐日时间轴

对每个站点，代码会先：

1. 按 `station_id + date` 排序
2. 合并同一天重复记录
3. 在该站点的起止日期之间按日补齐时间轴

这样做之后：

- 断档会显式变成 `NaN`
- 不会把相隔很多天的观测误当成连续时间点
- 即使同一天只观测到 `specific_conductance` 或只观测到 `turbidity`，该行也不会被整体丢弃

### 5.2 保留缺测信息

对每个原始动态变量，都会额外生成：

- `xxx_observed`

此外还会有：

- `is_original_observation`
- `is_gap_filled`
- `any_dynamic_observed`

所以模型能区分：

- 真实低值
- 缺失值
- 由时间轴补齐产生的空档

### 5.3 再做 lag / rolling / seasonal

对动态变量和目标变量本身，代码会生成：

- `lag_1 / lag_3 / lag_7 / ...`
- `roll_mean`
- `roll_std`

如果配置打开 `add_seasonal`，还会加：

- `doy_sin`
- `doy_cos`
- `month`

### 5.4 目标标签怎么构造

对每个目标，例如 `specific_conductance`：

- `target_conductance_h1`
- `target_turbidity_h1`

本质上就是：

- 用今天的可用信息
- 预测未来 `horizon` 的真实值

只有未来标签存在的样本才会保留。

## 6. sequence 路线怎么处理 dynamic 数据

如果模型走序列路线，也就是：

- `seq_tcn_tail`
- `seq_transformer_tail`

那么 `data.py` 会进一步把站点-日期表转换成：

- `lookback window -> sequence tensor`

### 6.1 sequence 输入包含什么

窗口里通常包括：

- 动态驱动变量
- 目标变量自身历史
- `is_gap_filled`
- 季节项

### 6.2 为什么断续站点也能处理

因为序列窗口是在**补齐后的日时间轴**上切出来的。

窗口里缺失值不会被偷偷抹平，而是会：

- 保留为 `NaN`
- 再配一个显式 `mask`

所以序列模型知道哪些时间步是真实观测，哪些只是补齐后没有观测的空位。

---

## 7. 输出层现在是什么结构

### 7.1 现在不再学习独立 point head

我们之前讨论过：

- `point head`
- `quantile head`
- `event head`

三者并列时，容易出现一致性冲突。

所以现在的实现做了更统一的取舍：

- 只学习 `quantile head`
- 和 `event head`
- `point prediction` 统一由 `q0.5` 导出

### 7.2 每个目标的输出内容

对每个目标变量，模型都会输出：

1. `point`
   - 实际来自 `q0.5`
2. `quantiles`
   - 例如 `0.1 / 0.5 / 0.9 / 0.95 / 0.99`
3. `exceedance_probability`
   - 预测超过设定阈值的概率

### 7.3 当前损失函数

核心损失包括：

- `quantile loss`
- `binary cross-entropy`
- 非负边界惩罚

其中高分位可以通过 `quantile_weights` 给更高权重。

## 8. 图相关模块现在怎么组织

这一部分是这次重构最关键的变化。

### 8.1 邻居统计路线仍然保留

`hydrotail/graph.py` 仍然会基于静态属性建立相似站点关系，并生成：

- `graph_neighbor_mean_xxx`
- `graph_neighbor_mean_distance`
- `graph_neighbor_count`

这条路线的本质是：

- **图启发特征工程**
- 不是 GNN

### 8.2 GNN 不再是独立模型

之前的设计是：

- `gnn_tail` 作为一个并列模型

现在已经改成：

- `torch_tail` 内部的 `graph_backend`

也就是说，图处理方式不再体现在“模型名字”上，而体现在：

- `models.torch_tail.graph_backend`

可选值有：

- `none`
- `neighbor_stats`
- `gnn`

### 8.3 三种 graph backend 的区别

#### `none`

- 不使用邻居统计特征
- 也不使用图传播

#### `neighbor_stats`

- 使用 `graph.py` 预先算好的邻居统计特征
- 然后把它们和其它表格特征一起送进 MLP

#### `gnn`

- 不再把 `graph_neighbor_*` 统计量送进表格特征
- 直接使用站点相似图 `edge_df`
- 在 `TorchTailModel` 内部做同日子图传播

### 8.4 当前 `gnn` 后端具体怎么工作

现在的 `gnn` 后端逻辑是：

1. 仍然以站点-日期表格特征作为节点特征
2. 对每个日期，取当天活跃站点构成一个子图
3. 图结构来自静态相似图 `edge_df`
4. 在这个同日子图上做轻量图卷积
5. 再接 quantile head 和 event head

也就是说它不是时空 GNN，而是：

- **时间信息来自 lag / rolling / seasonal**
- **空间信息来自同日图传播**

这是一个非常适合当前阶段的版本，因为它正好能回答：

> 相比“把邻居关系预先压成统计特征”，直接在图上传播是否真的有增益？

## 9. 当前有哪些模型可以跑

### 9.1 表格基线

- `linear_tail`
- `gbdt_tail`

它们都输出：

- `point = q0.5`
- quantiles
- exceedance probability

### 9.2 主深度表格模型

- `torch_tail`

它是当前最重要的主模型，因为它现在已经内置了：

- `graph_backend = none`
- `graph_backend = neighbor_stats`
- `graph_backend = gnn`

### 9.3 序列模型

- `seq_tcn_tail`
- `seq_transformer_tail`

它们仍然专注于：

- `lookback window`
- 时序编码
- 多任务尾部输出

目前图后端还没有并到序列模型里。

## 10. 切分和分析现在怎么组织

### 10.1 基本切分

默认主场景是：

- `unseen_station_and_future`

基础集合有：

- `train`
- `valid`
- `test`

### 10.2 额外四类分析切片

现在代码还会额外保存：

- `train_station_early`
- `train_station_late`
- `test_station_early`
- `test_station_late`

这使得结果分析不再只是“train/test 两行表”，而是能明确回答：

- 熟站点早期表现如何
- 熟站点后期表现如何
- 新站点早期表现如何
- 新站点后期表现如何

这对你想做的 `unseen_station_and_future` 分析非常重要。

## 11. `train.py` 现在怎么跑完整实验

`hydrotail/train.py` 现在做的事情是：

1. 读配置
2. 读数据
3. 生成表格特征和图边
4. 根据策略切分数据
5. 计算尾部阈值
6. 计算训练样本权重
7. 视模型类型分别训练
8. 对所有评估切片分别输出：
   - metrics
   - predictions
   - model.bin

如果是 `torch_tail`，它现在会同时拿到：

- `feature_cols`
- `edge_df`
- `station_col`
- `date_col`

从而决定内部到底走：

- 普通 MLP
- 还是 GNN backend

## 12. 当前 smoke test 在验证什么

`hydrotail/smoke_test.py` 现在不再只跑一套实验。

它会跑两套：

1. `outputs/smoke_test_neighbor_stats`
   - `torch_tail.graph_backend = neighbor_stats`
   - 同时带上 baseline 和 sequence 模型
2. `outputs/smoke_test_gnn`
   - `torch_tail.graph_backend = gnn`
   - 专门验证新的 GNN 后端路径

这能保证：

- 邻居统计路线没坏
- GNN 后端能训练、预测、存结果
- 四类分析切片仍然能正常输出

## 13. 当前这版的优点

### 13.1 结构比之前统一

现在“图信息怎么用”变成了主模型内部的一个可选设计，而不是再单独长出一个模型家族。

### 13.2 更适合做方法对照

你现在可以非常干净地比较：

- `torch_tail + none`
- `torch_tail + neighbor_stats`
- `torch_tail + gnn`

而不是比较三个名字和训练路径都不太一样的模型。

### 13.3 更贴近论文方法部分

这种写法更像：

> 同一个多任务尾部预测框架，在不同图信息注入方式下的比较

而不是：

> 一堆并列模型拼盘

---

## 14. 当前还没做到的事情

### 14.1 旧的 MATLAB timetable `.mat` 仍然不能直接喂给 loader

虽然当前项目已经有可直接读取的 WQ csv，但旧的 `.mat` 仍然还是 opaque MCOS timetable。

所以如果未来又只有 MAT，没有 csv，仍然需要先导出。

### 14.2 当前 GNN 还是轻量版本

现在的 `graph_backend = gnn`：

- 不是时空 GNN
- 不是 Graph Transformer
- 也还没有和 `TCN/Transformer` 序列编码器耦合

但它已经足够支持第一轮方法比较。

### 14.3 解释分析脚本还没有单独写出来

当前已经能保存足够多的预测结果，但还没有把：

- SHAP
- 区域失效分析
- 尾部事件归因分析

写成独立脚本。

---

## 15. 现在最自然的实验顺序

如果接下来继续推进，我建议这样做：

1. 先用当前 `dataset/WQ/SEC` 和 `dataset/WQ/Tur` 跑通真实数据的第一轮基线
2. 先跑：
   - `linear_tail`
   - `gbdt_tail`
   - `torch_tail` with `graph_backend=neighbor_stats`
3. 再跑：
   - `torch_tail` with `graph_backend=gnn`
4. 如果需要更强时序模型，再跑：
   - `seq_tcn_tail`
   - `seq_transformer_tail`
5. 重点分析：
   - `test_station_early`
   - `test_station_late`
   - 高值尾部指标
   - `neighbor_stats` vs `gnn`

这会比继续堆新的复杂模块更稳，也更容易形成一版像样的实验主线。
